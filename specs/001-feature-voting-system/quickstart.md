# Quickstart & Validation: Feature Voting System

**Plan**: [plan.md](./plan.md) | **Contract**: [contracts/openapi.yaml](./contracts/openapi.yaml) | **Data model**: [data-model.md](./data-model.md)

A run-and-validate guide proving the feature works end to end. It does **not**
contain implementation code — see `tasks.md` (generated by `/speckit-tasks`) and
the implementation phase for that.

---

## Prerequisites

- Docker + Docker Compose (pinned versions)
- Go 1.23+ (only for running tests locally outside containers)
- `make`, `curl`, `jq` for the manual flow below

## Configuration

Copy the example env and adjust as needed (12-factor; never commit real secrets):

```bash
cp deploy/.env.example deploy/.env
```

Required keys (documented in `.env.example`): `DATABASE_URL`, `REDIS_URL`,
`JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, `JWT_ACCESS_TTL`, `JWT_REFRESH_TTL`,
`TRENDING_RECOMPUTE_INTERVAL`, `PORT`.

## Bring the stack up

```bash
docker compose -f deploy/docker-compose.yml up --build
```

This starts `postgres`, `redis`, runs `golang-migrate` (one-shot, before the app),
then `app`. Wait for readiness:

```bash
curl -fsS http://localhost:8080/readyz   # 200 when Postgres + Redis are reachable
```

## End-to-end validation scenarios

Each maps to user stories / functional requirements / success criteria in
[spec.md](./spec.md). Endpoints and payloads are defined in the OpenAPI contract.

### Scenario A — Submit a request (US1 · FR-002, FR-003 · SC-001)

1. Register and capture a token:
   ```bash
   TOKEN=$(curl -fsS -X POST localhost:8080/auth/register \
     -H 'content-type: application/json' \
     -d '{"email":"alice@example.com","password":"correct horse battery"}' | jq -r .access_token)
   ```
2. Submit a request, then list it:
   ```bash
   curl -fsS -X POST localhost:8080/requests -H "authorization: Bearer $TOKEN" \
     -H 'content-type: application/json' \
     -d '{"title":"Dark mode","description":"Please add a dark theme."}'
   ```
   **Expected**: `201`, body has `vote_count: 0` and `author_id` = Alice.
3. Submit with an empty title → **expected** `400` with a `validation_failed`
   message; nothing saved (US1-2, FR-003).
4. Repeat step 2 with no `Authorization` header → **expected** `401` (US1-3, FR-001).

### Scenario B — Browse with pagination (US2 · FR-004 · SC-007)

1. Seed >1 page of requests (loop the submit call ~25 times).
2. First page:
   ```bash
   curl -fsS "localhost:8080/requests?sort=new&limit=20" -H "authorization: Bearer $TOKEN"
   ```
   **Expected**: 20 `items` + a non-null `next_cursor`.
3. Next page using the returned cursor → **expected**: the following items, **no
   duplicates and no skips** across the boundary (SC-007).
4. Request a page past the end → **expected**: empty `items`, `next_cursor: null`
   (edge case "pagination boundaries"), not an error.

### Scenario C — Upvote rules (US3 · FR-006–FR-010)

1. Bob (a second user) upvotes Alice's request:
   ```bash
   curl -fsS -X PUT localhost:8080/requests/$ID/vote -H "authorization: Bearer $BOB"
   ```
   **Expected**: `200`, `vote_count: 1`, `viewer_has_voted: true` (FR-006, FR-010).
2. Bob repeats the same PUT → **expected**: still `vote_count: 1` (idempotent,
   FR-007, US3-2).
3. Alice upvotes her **own** request → **expected**: `403` `forbidden_self_vote`,
   count unchanged (FR-008, US3-3).
4. Bob removes his vote (`DELETE …/vote`) → **expected**: `vote_count: 0`,
   `viewer_has_voted: false`; he may vote again later (FR-009, US3-4).
5. Vote on a random non-existent UUID → **expected**: `404`, no orphan vote
   (FR-018, edge case "vote on a removed request").

### Scenario D — Top vs Trending rankings (US4 · FR-011–FR-013 · SC-006)

1. Arrange data: request X with many **old** votes, request Y with the same number
   of **recent** votes (integration tests seed `created_at`/vote times directly).
2. `GET /requests?sort=top` → **expected**: ordered by `vote_count` desc; ties
   broken deterministically so paging is stable (FR-011, FR-013).
3. Wait one `TRENDING_RECOMPUTE_INTERVAL`, then `GET /requests?sort=trending` →
   **expected**: Y (recent) ranks above X (FR-012, SC-006).

### Scenario E — Optimistic UI + conditional polling (US5 · FR-014–FR-016 · SC-005)

This is primarily a client behavior; the server side validated here is the
revalidation contract:

1. `GET /requests?sort=top` and capture the `ETag`.
2. Re-request with `If-None-Match: <etag>` and no intervening change →
   **expected**: `304 Not Modified` (FR-015, cheap polling).
3. Cast a vote, then re-request with the old ETag → **expected**: `200` with a new
   ETag and updated counts, reconciling within ≤60 s (SC-005). Optimistic
   increment + rollback on a rejected vote is exercised by client/e2e tests.

### Scenario F — Viral spike correctness (Edge case · FR-007, FR-017 · SC-003, SC-004)

Run the integration/load test (testcontainers, real Postgres + Redis):

```bash
go test -tags=integration -race ./tests/integration/ -run Spike
```

**Expected**: 1,000 distinct users each upvote one request within a minute →
final `vote_count == 1000` exactly (no lost/double counts), every duplicate attempt
is a no-op, and browse/vote stay responsive (SC-003, SC-004). A reconciliation pass
confirms `vote_count == COUNT(votes)`.

## Automated test suite

```bash
make lint            # gofmt -s, go vet, staticcheck, golangci-lint (gosec)
make test            # go test -race ./...  (unit)
make test-integration   # go test -tags=integration ./...  (testcontainers: real PG + Redis)
make vuln            # govulncheck ./...
```

CI runs the same gates in order: `golangci-lint → test -race → govulncheck →
docker build`. All constitution Pre-Commit Quality Gates must pass before merge.

## Tear down

```bash
docker compose -f deploy/docker-compose.yml down        # keep named volumes
docker compose -f deploy/docker-compose.yml down -v      # also drop data (destructive)
```

> Named volumes persist Postgres + Redis data across restarts; `-v` deletes them.
